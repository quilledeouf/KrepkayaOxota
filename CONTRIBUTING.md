# Как мы работаем — Крепкая Охота

Документ для команды разработки. Описывает ветки, правила коммитов и порядок
внесения изменений. Цель — чтобы любой участник мог подключиться к коду без
путаницы.

---

## 1. Ветвление (упрощённый Git Flow)

| Ветка | Назначение | Кто пушит напрямую |
|-------|-----------|--------------------|
| `main` | Только проверенные релизы. Всегда «зелёная». | Никто (только merge через PR) |
| `develop` | Интеграция фич. Здесь собирается следующий релиз. | Через PR из `feature/*` |
| `feature/*` | Одна ветка = одна задача из [бэклога](docs/BACKLOG.md). | Автор фичи |

Дополнительно при необходимости:

| Ветка | Назначение |
|-------|-----------|
| `fix/*` | Исправление бага. |
| `hotfix/*` | Срочное исправление прямо в `main`. |

### Схема

```
main  ───────●───────────────────●─────►   (релизы: v0.1, v0.2 …)
              \                  /
develop ───●───●────●────●──────●─────────►  (интеграция)
            \     /    \      /
feature/F1   ●──●    feature/F3 ●──●
```

### Жизненный цикл задачи

```bash
git checkout develop
git pull
git checkout -b feature/F1-interactive-map   # имя = ID задачи + краткое описание
# ... работаем, коммитим ...
git push -u origin feature/F1-interactive-map
# открываем Pull Request feature/F1-* → develop, ревью, merge
```

## 2. Именование веток

```
feature/<ID>-<короткое-описание>     feature/F1-interactive-map
fix/<что-чиним>                      fix/season-toggle-reset
hotfix/<что-чиним>                   hotfix/map-crash-no-data
```

ID берём из [технического бэклога](docs/BACKLOG.md) (DEV-01 … DEV-24) или из
функциональных требований (F1 … F8).

---

## 3. Сообщения коммитов (Conventional Commits)

```
<тип>: <краткое описание в повелительном наклонении>

[необязательное тело: что и зачем]
```

Типы:

| Тип | Когда |
|-----|-------|
| `feat` | новая функциональность |
| `fix` | исправление бага |
| `docs` | документация |
| `style` | форматирование, без изменения логики |
| `refactor` | переработка кода без новой функциональности |
| `test` | тесты |
| `chore` | инфраструктура, конфиги, зависимости |

Примеры:

```
feat: добавить переключатель сезона на карте
fix: карточка региона не закрывалась по Esc
docs: описать API-контракт для /regions
```

## 4. Pull Request — чек-лист

- [ ] Ветка названа по правилам, базируется на актуальном `develop`.
- [ ] Изменения соответствуют одной задаче (не «всё сразу»).
- [ ] Прототип запускается (`python -m http.server` в `frontend/`).
- [ ] Нет ошибок в консоли браузера.
- [ ] Обновлена документация, если менялся контракт/структура.
- [ ] Назначен ревьюер (второй разработчик).

## 5. Структура кода фронтенда

| Файл | Ответственность |
|------|-----------------|
| `frontend/js/data.js` | Встроенные демо-данные (заменяется реальным API на этапе 2) |
| `frontend/js/api.js` | Слой доступа к данным. Единая точка, через которую UI берёт данные |
| `frontend/js/map.js` | Инициализация и логика карты Leaflet |
| `frontend/js/app.js` | Состояние приложения, навигация, обработчики UI |

Правило: **UI не лезет в данные напрямую** — только через `api.js`. Тогда при
появлении реального бэкенда достаточно переписать `api.js`, не трогая UI.

---

## 6. Code style

- Отступ — 2 пробела, кодировка UTF-8, переводы строк LF.
- Имена переменных/функций — `camelCase`, на английском.
- Комментарии — на русском, поясняют «зачем», а не «что».
- Никаких секретов в коде (см. `.gitignore`).
